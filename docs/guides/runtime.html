Runtime

LatticePHP supports multiple PHP runtimes. Your application code does not change between environments -- the runtime adapter handles the differences.

Runtime Options

Runtime Status Use Case
PHP-FPM Baseline Standard hosting, shared environments, simplest deployment
RoadRunner First-class High-performance long-running workers, recommended for production
OpenSwoole Experimental Coroutine-based async, advanced use cases

PHP-FPM (Baseline)

PHP-FPM is the default runtime. No additional packages required.

Development Server

php lattice serve
# Starts PHP built-in server on http://localhost:8000

php lattice serve --port=3000
# Custom port

Production with Nginx

server {
    listen 80;
    server_name api.example.com;
    root /var/www/app/public;
    index index.php;

    location / {
        try_files $uri /index.php$is_args$args;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/run/php/php8.4-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
        include fastcgi_params;
    }
}

PHP-FPM boots and destroys the application on every request. Simple, predictable, no state leakage. The trade-off is startup overhead on every request.

RoadRunner (First-Class)

RoadRunner keeps your PHP application in memory between requests. The framework boots once, then handles thousands of requests without re-bootstrapping.

Installation

composer require lattice/roadrunner

Key Classes

All classes are in the Lattice\RoadRunner namespace:

RoadRunnerConfig -- Worker configuration:

use Lattice\RoadRunner\RoadRunnerConfig;

$config = new RoadRunnerConfig(
    httpWorkers: 4,
    grpcWorkers: 4,
    maxMemory: 128,          // MB per worker
    resetProviders: [        // Providers to reset between requests
        CacheServiceProvider::class,
    ],
);

// Or from array (e.g., config file)
$config = RoadRunnerConfig::fromArray([
    'httpWorkers' => 8,
    'maxMemory' => 256,
]);

WorkerLifecycle -- Hooks into the worker lifecycle:

use Lattice\RoadRunner\WorkerLifecycle;

$lifecycle = new WorkerLifecycle();

$lifecycle->onStartup(function (): void {
    // Called once when the worker process starts
    // Initialize connections, warm caches
});

$lifecycle->onRequest(function (): void {
    // Called after each request completes
    // Useful for clearing per-request state
});

$lifecycle->onDrain(function (): void {
    // Called when the worker is draining (preparing to stop)
    // Finish in-flight work, close connections gracefully
});

$lifecycle->onShutdown(function (): void {
    // Called when the worker is stopping
    // Final cleanup
});

ContainerResetter -- Clears container state between requests:

use Lattice\RoadRunner\ContainerResetter;

$resetter = new ContainerResetter();

$resetter->addResetCallback(function (): void {
    // Clear per-request singletons, flush resolved instances
});

// Called after each request:
$resetter->reset($container); // Runs callbacks then calls $container->reset()

MemoryGuard -- Prevents memory leaks from crashing workers:

use Lattice\RoadRunner\MemoryGuard;

$guard = new MemoryGuard();

$guard->getCurrentUsageMb(); // e.g., 45.3
$guard->check(128);          // true if usage >= 128 MB

When MemoryGuard::check() returns true, the worker stops and RoadRunner spawns a replacement. This prevents memory leaks from degrading performance over time.

GracefulShutdown -- Handles SIGTERM/SIGINT:

use Lattice\RoadRunner\GracefulShutdown;

$shutdown = new GracefulShutdown();
$shutdown->register(); // Register signal handlers (no-op on Windows)

// In the worker loop:
if ($shutdown->shouldStop()) {
    // Finish current request, then exit
}

RoadRunnerHttpWorker

The RoadRunnerHttpWorker (Lattice\RoadRunner\Http\RoadRunnerHttpWorker) ties everything together:

use Lattice\RoadRunner\Http\RoadRunnerHttpWorker;

$worker = new RoadRunnerHttpWorker(
    container: $container,
    containerResetter: $resetter,
    lifecycle: $lifecycle,
    memoryGuard: $guard,
    gracefulShutdown: $shutdown,
    config: $config,
);

$worker->start();  // Enter worker loop
// After each request, afterRequest() runs automatically:
// 1. Trigger request callbacks
// 2. Reset container state
// 3. Check memory limit (stop if exceeded)
// 4. Check graceful shutdown signal

RoadRunner .rr.yaml

version: "3"

server:
  command: "php worker.php"

http:
  address: "0.0.0.0:8080"
  pool:
    num_workers: 4
    max_jobs: 1000        # Restart worker after N requests
    allocate_timeout: 10s
    destroy_timeout: 10s

logs:
  mode: production
  level: info

Common Pitfalls

  1. Static state persists between requests. Clear static properties in reset callbacks or avoid them entirely.
  2. Database connections may timeout. Configure connection pool timeouts in your database config.
  3. File handles leak. Close all file handles in request callbacks.
  4. Memory grows gradually. Set maxMemory to catch leaks before they cause OOM kills.

OpenSwoole (Experimental)

OpenSwoole provides coroutine-based concurrency. This runtime is experimental and not recommended for production without thorough testing.

composer require lattice/openswoole

OpenSwoole brings true async I/O to PHP -- database queries, HTTP calls, and file operations can run concurrently within a single worker. The trade-off is more complex debugging and potential issues with non-coroutine-safe extensions.

Docker Deployment

PHP-FPM Dockerfile

FROM php:8.4-fpm-alpine

RUN apk add --no-cache icu-dev \
    && docker-php-ext-install intl pdo_mysql opcache

COPY --from=composer:latest /usr/bin/composer /usr/bin/composer
COPY . /var/www/app
WORKDIR /var/www/app

RUN composer install --no-dev --optimize-autoloader
RUN php lattice compile

EXPOSE 9000
CMD ["php-fpm"]

RoadRunner Dockerfile

FROM php:8.4-cli-alpine

RUN apk add --no-cache icu-dev \
    && docker-php-ext-install intl pdo_mysql opcache pcntl

COPY --from=composer:latest /usr/bin/composer /usr/bin/composer
COPY . /var/www/app
WORKDIR /var/www/app

RUN composer install --no-dev --optimize-autoloader
RUN php lattice compile
RUN vendor/bin/rr get-binary

EXPOSE 8080
CMD ["./rr", "serve", "-c", ".rr.yaml"]

Choosing a Runtime

Factor PHP-FPM RoadRunner OpenSwoole
Setup complexity Low Medium High
Performance Good Excellent Excellent
Memory efficiency High (per-request) Medium (pooled) Medium (pooled)
State management No concerns Must reset state Must reset state
Debugging Standard tools Standard tools Coroutine-aware needed
Production readiness Proven Proven Experimental

Start with PHP-FPM. Move to RoadRunner when you need lower latency and higher throughput. Consider OpenSwoole only for specific async use cases.
